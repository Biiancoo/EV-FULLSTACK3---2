================================================================================
1. BACKEND - NUEVO ARCHIVO: AiRecommendationService.java
================================================================================

Ubicacion:
  SmartLogix/inventory-service/src/main/java/.../service/AiRecommendationService.java

Que hace:
  Es el CEREBRO de la IA. Este servicio se encarga de:
  
  a) Tomar todos los productos del inventario desde la base de datos (H2).
  b) Construir un PROMPT (texto de instrucciones) con los datos del inventario.
  c) Enviar ese prompt a la API de Google Gemini (via HTTP POST).
  d) Recibir la respuesta en JSON, limpiarla y convertirla a objetos Java.
  e) Si Gemini falla o no hay API key configurada, usar el sistema de reglas
     como fallback (respaldo) automaticamente.

Partes clave del codigo:

- `geminiApiKey`: lee la variable de entorno GEMINI_API_KEY. Si esta vacia,
  salta al fallback.

- `buildPrompt()`: crea el texto que le enviamos a la IA. Le dice:
  "Eres un asistente de logistica. Analiza estos productos y dime que pedir
   y en que cantidad. Responde SOLO con JSON."
  
- `callGemini()`: hace la llamada HTTP a:
  https://generativelanguage.googleapis.com/v1beta/models/gemini-2.0-flash:generateContent
  
- `extractTextFromResponse()`: extrae el texto de la respuesta de Gemini
  (la respuesta viene anidada en "candidates" -> "content" -> "parts" -> "text").

- `fallbackService.getRecommendations()`: si algo falla, usa el sistema de
  reglas que ya existia (InventoryService.getRecommendations()).

  Se mantuvo el metodo `getRecommendations()` que YA habiamos creado antes,

  porque el sistema sigue respondiendo con recomendaciones validas.

6. BACKEND - NUEVO ARCHIVO: InventoryRecommendationResponse.java
================================================================================

Ubicacion:
  SmartLogix/inventory-service/src/main/java/.../dto/InventoryRecommendationResponse.java

Que hace:
  Es un DTO (Data Transfer Object) que define la estructura de la respuesta.
  Es la "caja" donde guardamos cada recomendacion. Contiene:
  - sku, productName, warehouseCode (identificacion del producto)
  - currentStock, reservedStock, reorderLevel (datos actuales)
  - recommendedQuantity (cuanto pedir - lo que calcula la IA)
  - urgency (CRITICAL, HIGH, MEDIUM, LOW)
  - reason (texto explicativo generado por la IA)
  - generatedAt (fecha/hora de la recomendacion)

  Tanto el sistema de reglas (fallback) como Gemini devuelven objetos
  con esta misma estructura. Asi el frontend no necesita saber de donde
  vino la respuesta.

RESUMEN: POR QUE ESTO ES "IA REAL"
================================================================================

Antes (solo reglas):
  El codigo Java tomaba decisiones con if/else. No habia inteligencia
  externa, solo algoritmos predefinidos.

Ahora (con Gemini):
  El codigo Java envia los datos a un modelo de lenguaje (LLM) de Google
  que fue entrenado con millones de datos. El modelo "piensa" y genera
  recomendaciones en texto natural. Es como tener un analista de
  logistica virtual que lee tu inventario y te dice que hacer.

  Si el modelo dice:
    "El stock del SKU-3001 es preocupante porque solo quedan 45 unidades
     y el nivel de reorden es 10. Dado que es un monitor de alta rotacion,
     recomiendo pedir 30 unidades para evitar quiebre de stock."
  
  Ese texto NO fue escrito por un if/else. Fue generado por la IA.

  Si Gemini no esta disponible, el sistema usa reglas como respaldo.
  Es robusto y profesional.
